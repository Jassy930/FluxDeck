# FluxDeck Native 拓扑 Sankey 主舞台重构设计

## 文档状态

- 状态：已实现
- 类型：设计文档，已完成落地
- 适用范围：`apps/desktop-macos-native`

## 背景

上一轮原生端 `Topology` 已完成三列 token flow 与链路内 model 分层，功能上满足了：

- `Tokens / Requests`
- `By Model / Total Only`
- `Top 3 / Top 5 / All`
- `Hot Paths + Model Mix`

但用户反馈当前画布“功能对了，视觉不对”，与参考的 VPN / Sankey 产品相差仍然明显。主要问题：

1. 节点卡片依旧过重，画面第一眼看到的是卡片而不是流带。
2. 流带体积感不足，更像“加粗连线”而不是连续的流量束。
3. 控制栏与画布争抢视觉中心，破坏了 Sankey 主舞台的连贯感。
4. 小流量链路可读性不足，最小宽度没有形成稳定规则。

因此这轮不是补功能，而是对 `Topology` 做一次明确的视觉重构，让页面真正进入 “Sankey 主舞台” 状态。

## 已确认产品决策

- 视觉方向采用 `方案 A：Sankey 主舞台`
- 三列结构继续保持：
  - `Entrypoints`
  - `Gateways`
  - `Providers`
- 不新增显式 `Models` 第四列
- 流带成为页面第一视觉语言
- 节点弱化为细窄锚点块
- 流带必须具备最小可读宽度
- 增加 hover 联动：
  - hover 流带时显示链路诊断 tooltip
  - hover 节点时高亮上下游链路与节点汇总

## 目标

1. 让用户第一眼先看到主干 token 流，而不是卡片和控件。
2. 让大中小流量链路都保持可读，尤其是小流量不能退化成噪声线。
3. 让当前三列结构更接近参考图的阅读节奏：
   - 看主干
   - 看分层
   - 看标签
4. 在不增加页面复杂度的前提下，用 hover 恢复节点和链路的诊断细节。

## 非目标

- 不新增拓扑专用后端接口
- 不引入拖拽布局、缩放、平移
- 不新增右侧详情抽屉
- 不把底部摘要改成第二个主舞台

## 视觉主张

### 1. 主画布

画布从“卡片容器”改为“连续主舞台”：

- 背景更纯净、更暗
- 中央流带区域成为视觉中心
- 三列只保留轻量锚点轨道感
- 标题区和控件区弱化，不抢主画布

### 2. 流带

流带不是粗线，而是封闭的带状体：

- 由封闭 Path 绘制
- 半透明实体填充表达体积
- model 层之间贴合堆叠，减少裂缝感
- 高负载主干允许轻微高光

厚度规则：

- 厚度主编码仍为 `total_tokens`
- 采用非线性缩放，防止头部链路过粗、尾部链路消失
- 所有流带遵守：
  - `rendered_width = max(min_readable_width, scaled_width(value))`

### 3. 最小宽度

这是这轮的强制约束：

- 即使链路 token 很小，也必须保持最低可读厚度
- 最小厚度仅保证可见，不改变相对排序
- 大流量仍然必须明显粗于小流量

### 4. 节点

节点从“卡片信息块”改为“流带挂靠锚点”：

- 更窄
- 更轻
- 更少文字
- 保留名称和一行极简摘要

默认不在节点上铺满：

- `cached`
- `error`
- 复杂诊断字段

这些细节转移给 hover 和底部摘要。

### 5. 控制栏

控制栏保留原有功能，但视觉弱化成轻悬浮条：

- `Tokens / Requests`
- `By Model / Total Only`
- `Top 3 / Top 5 / All`

要求：

- 更紧凑
- 更贴边
- 透明度更低
- 不再像一个独立大面板

### 6. 底部摘要

底部仍保留：

- `Hot Paths`
- `Model Mix`

但角色改成“读图结论区”：

- 不抢主画布
- 更紧凑
- 强调数字总结而不是大面积块面

## Hover 交互

### 1. Hover 流带

- 高亮当前链路
- 其它链路降透明度
- 显示 tooltip：
  - `from -> to`
  - `model`
  - `total tokens`
  - `requests`
  - `cached tokens`
  - `errors`

### 2. Hover 节点

- 高亮该节点所有上下游链路
- 非关联链路与节点降低透明度
- 显示节点汇总 tooltip：
  - `总 token`
  - `总 req`
  - `top model`
  - `error count`

## 数据与实现边界

本轮继续使用已有客户端聚合模型：

- `TopologyGraph`
- `TopologyEdge`
- `TopologyEdgeSegment`

不修改：

- 后端契约
- `AdminLog` 接口
- 统计聚合字段

本轮重点是：

- 视觉比例
- 流带绘制
- 节点弱化
- hover 状态机

## 验收标准

1. 页面第一眼主角变成流带而不是节点卡片。
2. 小流量链路在暗背景上仍然可见。
3. `All` 模式下不同 model 仍可区分，不出现大面积同色坍缩。
4. hover 能补回节点和链路诊断信息，不需要重新把节点做重。
5. 整体观感明显更接近参考图的 Sankey 主舞台。

## 实现结果

- 已保持三列结构：
  - `Entrypoints`
  - `Gateways`
  - `Providers`
- 已将主舞台改为以封闭流带为主体的 Sankey 画布：
  - 链路宽度抽为纯函数并应用最小可读厚度
  - `Tokens / Requests` 共用相同的最小宽度约束
- 已将节点降级为轻量锚点：
  - 主信息压缩为单行 token/request 摘要
  - `cached` / `err` 只保留极简次信息，详细诊断转移到 hover
- 已加入 hover 诊断：
  - hover 流带显示链路 tooltip
  - hover 节点显示节点汇总 tooltip
  - 非关联链路与节点应用降透明度语义
