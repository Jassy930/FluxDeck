# FluxDeck Native Traffic 趋势图柔化曲线设计

## 文档状态

- 状态：已确认
- 类型：设计文档，待进入实现
- 适用范围：`apps/desktop-macos-native`

## 背景

当前原生端 `Traffic` 页面中的 `Token Trend by Model` 主图已经切换为堆叠面积 + 边界折线的结构，但图形几何仍然使用折线段逐点连接。虽然线帽与转角已使用圆角描边，视觉上仍然会出现明显硬折，尤其在 token 峰值快速抬升或回落时，图形显得偏“硬”，与整体深色控制台的柔和玻璃质感不一致。

用户要求在保持监控可读性的前提下，将这张图改成更柔和的曲线。

## 目标

1. 将总量线与各模型边界线从直线段改为平滑曲线。
2. 让堆叠面积的上边界与折线使用同一套曲线几何，避免描边与填充割裂。
3. 保留 bucket 级 hover、tooltip、竖向高亮线与颜色映射，不改变图表信息结构。
4. 控制曲线张力，避免为追求柔和而制造视觉失真或虚假峰值。

## 非目标

- 不修改后端契约、bucket 聚合逻辑或时间密度。
- 不新增图例显隐、缩放、拖拽等交互。
- 不修改 `apps/desktop/` Web 桌面端。
- 不引入第三方图表库。

## 已确认方案

采用“中度柔化”方案：

1. 所有趋势折线改为三次贝塞尔平滑曲线。
2. 堆叠面积的上边界同步使用相同平滑算法。
3. 使用中等张力参数，让峰谷变圆，但保留监控图的节奏感。
4. 对控制点进行边界约束，避免陡升陡降时出现明显过冲。

## 方案对比

### 方案 A：轻柔化

- 仅将折线替换为平滑样条
- 面积层仍保留折线边界

优点：

- 变更最小
- 风险最低

缺点：

- 折线与填充几何不一致
- 局部会出现“线圆面折”的割裂感

### 方案 B：中度柔化

- 折线与面积上边界都使用共享平滑路径
- 对控制点施加张力与边界约束

优点：

- 柔和感明显提升
- 视觉统一
- 不会完全丢失监控图的紧凑感

缺点：

- 实现复杂度高于方案 A
- 需要额外测试避免路径越界

### 方案 C：仅柔化总量线

- 只有 `Total Tokens` 使用曲线
- 模型边界与面积保留折线

优点：

- 总量趋势更醒目

缺点：

- 整体风格不统一
- 与用户“折线图太硬”的诉求不完全匹配

## 几何策略

### 1. 统一路径构造

新增共享路径构造函数，例如：

- `smoothedLinePath(points:tension:)`
- `smoothedClosedAreaPath(upperPoints:lowerPoints:tension:)`

要求：

- 折线与面积共用同一套控制点推导逻辑
- 若点数不足两点，则回退到空路径或简单直线

### 2. 曲线算法

采用分段三次贝塞尔曲线：

- 每段控制点基于前一点、当前点、下一点、后一点推导
- `tension` 使用中间值，目标范围约 `0.35 ~ 0.45`
- 首尾点使用退化邻点，避免端点异常拉扯

该算法的目的不是无限平滑，而是在深色监控面板里消除硬折。

### 3. 过冲约束

对每段控制点的 `y` 值做钳制：

- 不高于该段相邻点的局部上界太多
- 不低于该段相邻点的局部下界太多
- 面积图上边界不能越过总量范围
- 面积闭合下边界仍按真实累计值回收，不做插值猜测

实现重点：

- 防止贝塞尔在尖峰附近画出不存在的二次峰值
- 防止曲线跌破下层堆叠边界

### 4. 视觉保留

以下元素保持原有行为：

- 总量线仍为最亮、最粗的轮廓线
- 各模型颜色与图例映射不变
- hover 命中仍按 bucket 索引工作，不跟随插值点
- tooltip 仍显示真实 bucket 数据，不显示曲线插值结果

## 测试策略

由于 SwiftUI `Path` 很难直接做逐像素断言，本次测试重点放在“几何辅助函数可验证”的层面：

1. 新增纯函数，输出平滑段控制点或路径片段描述。
2. 对 2 点、3 点、尖峰、谷底、平台等场景写单测。
3. 断言：
   - 路径段数量正确
   - 首尾点与原始点一致
   - 控制点落在约束范围内
   - 不会因单点峰值出现明显越界

对于视图层：

- 保持现有 `TrafficTrendChart` 相关测试不退化
- 如有必要，补一个针对辅助函数的 XCTest
