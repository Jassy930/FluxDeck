# FluxDeck 原生正式分发设计（待定方案）

## 文档状态

- 状态：待定
- 类型：设计文档，非实现计划
- 适用范围：面向普通 macOS 用户的一键安装分发
- 备注：本文档用于固化当前阶段的推荐分发方向，不代表立即进入实现

## 目标

为 FluxDeck 设计一条面向普通 macOS 用户的正式分发路线，使用户能够通过一次安装获得：

- 可直接打开的原生桌面应用
- 自动可用的本地后台服务
- 不依赖命令行的基础使用体验

## 背景

根据当前仓库现状，FluxDeck 的实际运行形态是：

- `fluxd`：本地后台服务
- `FluxDeckNative.app`：原生 macOS 控制台
- `fluxctl`：可选 CLI

当前原生壳默认连接本机 `127.0.0.1:7777`，并不会自行启动或托管 `fluxd`。因此它还不具备“单独作为普通用户桌面应用直接分发”的条件。

## 已确认的产品边界

本设计基于以下已确认约束：

- 目标用户是普通 macOS 用户，而不是开发者
- 安装体验目标是一键安装使用
- 后台服务应常驻运行，不随 App 退出而退出
- 首版正式分发不支持连接远端或其他机器上的 Admin API
- 首版正式分发仅支持本机使用场景

## 非目标

当前方案明确不覆盖以下内容：

- 立即开始实现
- Web 桌面壳作为主分发路线
- 系统级 `LaunchDaemon`
- 企业级多用户共享实例
- 自动更新
- 远端 Admin API
- 用户自定义后台服务地址

## 方案对比

### 方案 A：PKG + 每用户 LaunchAgent + 原生 App

形态：

- 用户下载安装一个经过签名与公证的 `FluxDeck.pkg`
- 安装器将 `FluxDeck.app` 安装到 `/Applications`
- 安装器同时在当前用户域安装 `fluxd` 与 `LaunchAgent`
- `fluxd` 由 `launchd` 负责常驻运行
- `FluxDeck.app` 仅作为本机控制台

优点：

- 最符合“普通用户一键安装”的目标
- 服务生命周期清晰，进程托管边界稳定
- App 与后台职责明确
- 便于后续做升级、卸载、诊断与支持

缺点：

- 需要安装器、签名、公证、服务注册与升级流程
- 实现和验证成本高于开发者分发

### 方案 B：单个 App，首次启动自注册后台服务

形态：

- 用户下载并打开单个 App
- App 在首次启动时把 `fluxd` 和 `LaunchAgent` 写入用户目录并注册

优点：

- 下载入口简单
- 表面上更接近常见拖拽式安装体验

缺点：

- 首次启动逻辑过重
- 故障恢复复杂
- 后台服务注册、重试与回滚更难维护
- 不如安装器路径稳定

### 方案 C：系统级守护进程 + App 控制台

形态：

- 通过系统级安装器部署后台服务
- App 仅连接系统级服务

优点：

- 服务管理能力更强
- 适合企业或多用户场景

缺点：

- 权限、安装、卸载复杂度最高
- 对当前项目阶段明显过重
- 不符合当前最小可行正式分发目标

## 当前推荐方向

推荐将 **方案 A：PKG + 每用户 LaunchAgent + 原生 App** 作为当前阶段的主设计方向。

原因：

- 它最贴合已经确认的产品边界
- 它不要求用户理解命令行、服务地址或开发者工作流
- 它保留了独立后台服务的架构优势
- 它比系统级守护进程更轻，比 App 首启自注册更稳定

需要强调的是：这里的“推荐”代表当前待定方案，而不是已经立项执行。

## 目标安装后形态

建议的安装后目录结构如下：

- `/Applications/FluxDeck.app`
- `~/Library/Application Support/FluxDeck/bin/fluxd`
- `~/Library/Application Support/FluxDeck/data/fluxdeck.db`
- `~/Library/Application Support/FluxDeck/logs/`
- `~/Library/LaunchAgents/com.fluxdeck.fluxd.plist`

设计意图：

- App 放在标准应用目录
- 后台二进制、数据和日志放在用户标准目录
- 不继续沿用开发阶段的 `~/.fluxdeck`
- 每个登录用户拥有独立实例与独立数据

## 服务生命周期设计

`fluxd` 被定义为“每用户常驻的本地后台服务”：

- 安装后注册 `LaunchAgent`
- 用户登录后自动拉起
- 异常退出后由 `launchd` 自动重启
- 固定监听 `127.0.0.1:7777`
- `FluxDeck.app` 不直接托管 `fluxd` 主进程

这意味着：

- App 退出不影响后台服务
- 后台服务故障恢复由系统托管机制优先处理
- App 的职责是控制台与诊断入口，而不是进程管理器

## App 角色收敛

正式分发后的 `FluxDeck.app` 应从“可配置 API 客户端”收敛为“本机 FluxDeck 控制台”。

建议保留：

- 服务状态展示
- 健康检查
- 服务重启入口
- 诊断信息查看与导出
- Provider / Gateway / Logs / Traffic 等主工作区

建议移除或仅保留给开发模式：

- `Admin URL` 手动输入
- 远端 Admin API 切换
- 任意地址重置与编辑
- 面向开发者的通用网络调试入口

## 故障恢复模型

建议首版只定义少数固定故障类型，并提供受控恢复动作：

### 1. 服务未运行

- 表现：App 无法连接本机 Admin API
- 动作：重试、重启服务、查看诊断

### 2. 端口不可用或冲突

- 表现：本机 `127.0.0.1:7777` 不可正常绑定
- 动作：显示明确诊断信息，不允许用户改地址规避

### 3. 数据目录或数据库异常

- 表现：`fluxd` 启动失败或迁移失败
- 动作：显示数据目录位置、日志位置、诊断摘要

### 4. App 与服务版本不匹配

- 表现：服务已运行，但 API 能力与 App 不匹配
- 动作：提示重新安装匹配版本，不继续静默运行

## 健康检查与诊断

为支撑正式分发场景，建议后续引入一个稳定健康接口，用于：

- App 启动时检测后台服务是否可用
- 展示服务版本与运行状态
- 判断 App 与服务版本是否匹配
- 为恢复页提供结构化诊断信息

建议诊断信息至少覆盖：

- App 版本
- `fluxd` 版本
- `LaunchAgent` 注册状态
- 后台进程存在性
- 本地端口连通性
- 数据目录与日志目录路径
- 最近一次启动错误摘要

## 打包与发布策略

建议首版正式分发只面向普通用户暴露一个正式入口：

- `FluxDeck.pkg`

同时保留一个面向高级用户的附加制品：

- `fluxctl` 单独压缩包

首版不建议同时暴露多个主安装入口，例如 `.dmg`、`.zip`、CLI、开发版 App 等，以免破坏“一键安装”的产品目标。

## 签名与公证要求

如果后续进入正式分发实现，建议必须满足：

- `FluxDeck.app` 完整签名
- 内嵌 `fluxd` 二进制签名
- `FluxDeck.pkg` 安装器签名
- Apple notarization
- 安装包完成 `staple`

在这些能力补齐前，不建议把方案称为“正式面向普通用户分发”。

## 版本策略

建议统一管理三层版本：

- 产品版本：对用户可见
- App 构建号：用于安装与排障
- `fluxd` 版本：与 App 强绑定

关键原则：

- `FluxDeck.app` 与 `fluxd` 必须同版本发布
- 升级通过重新安装新版 PKG 完成
- 默认保留用户数据与数据库
- 发现版本不匹配时，优先提示重新安装匹配版本

## 方案的阶段性判断

当前阶段的判断如下：

- 该方案适合作为中期正式分发方向
- 该方案当前仍是待定设计，不进入实现
- 在项目主线仍快速演进时，应允许继续调整目录、生命周期和安装策略
- 后续如果产品定位变化，例如重新支持远端 Admin API，需要重新审视本设计

## 当前待定项

以下两项保留为明确的待定议题，在真正进入实现前继续讨论，不在当前文档中拍板：

### 待定项 1：LaunchAgent 的安装、升级与回滚策略

需要后续明确：

- 安装器首次注册 `LaunchAgent` 的具体时机
- 升级时如何安全覆盖 `plist` 与后台二进制
- 升级失败时是否需要自动回滚
- 用户手动损坏或删除 `LaunchAgent` 后，App 应如何提示与修复

### 待定项 2：App 内“服务不可用恢复页”与“诊断页”的信息边界

需要后续明确：

- 恢复页显示哪些状态、哪些操作、哪些错误摘要
- 诊断页显示到什么粒度才既足够排障又不把普通用户淹没在技术细节中
- 面向普通用户的信息与面向支持排障的信息如何分层
- 是否需要单独的“导出诊断包”入口

## 后续讨论主题

在真正进入实现前，建议继续围绕以下问题迭代方案：

1. `fluxd` 的安装位置是否最终固定为 `Application Support`
2. 健康接口的最小字段集合
3. 安装、升级、卸载时的数据保留策略
4. 首版是否需要额外提供支持排障的 CLI 附件

## 结论

截至 2026-03-12，FluxDeck 面向普通 macOS 用户的正式分发待定主方案为：

**签名并公证的 `FluxDeck.pkg` + 每用户 `LaunchAgent` 常驻 `fluxd` + 只连接本机服务的 `FluxDeck.app` 控制台。**

该方案当前用于指导后续讨论与收敛，不作为立即实施承诺。
