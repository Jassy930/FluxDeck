# 计划文档生命周期规则

`docs/plans/` 只承载可执行的设计、调查与实施计划，不承担长期产品说明或一次性运行记录。

## 目录职责

### `active/`

只放当前仍在推进、尚未完成、仍需要继续执行或继续决策的文档。

允许类型：

- 正在执行的实施计划
- 尚未定案的调查文档
- 已确认会继续推进的设计文档

不应放入：

- 已完成且已合入主线的计划
- 只记录本地一次性操作的说明
- 主要内容已经转成“实施结果”的总结文档

### `completed/`

放已完成、已验证、已合入主线，且后续主要用于追溯的计划与设计文档。

### 根目录

默认不存放普通计划文档。

仅允许以下内容长期存在：

- 本说明文件 `README.md`
- 目录占位文件
- 经过明确约定的特殊索引页

如果某份普通计划文档仍停留在根目录，说明它尚未完成归档整理。

当前可执行检查入口：

```bash
./scripts/check_docs_plan_layout.sh
```

该脚本会检查：

- `docs/plans/` 根目录是否残留普通计划文档
- `docs/plans/active/` 是否出现明显完成态信号

## 推荐命名

- 设计：`YYYY-MM-DD-<topic>-design.md`
- 调查：`YYYY-MM-DD-<topic>-investigation.md`
- 实施计划：`YYYY-MM-DD-<topic>.md`
- 问题清单 / backlog：`YYYY-MM-DD-<topic>-backlog.md`

同一主题建议保持可配对关系：

- 先设计，再实施
- 调查若转为长期事项，应回链到 backlog 或后续设计

## 最小交叉引用要求

每份计划至少应能回答三个问题：

1. 为什么做
2. 做到什么算完成
3. 完成后该去哪里找结果

因此建议每份计划至少链接到：

- 对应的契约 / 运行手册 / 测试入口
- 对应的 `docs/progress/` 记录
- 若属于长期治理问题，链接对应 backlog 条目

## 归档时机

满足以下条件后，应从 `active/` 迁出：

- 文档目标已经完成，或调查已经结束
- 对应实现和记录已进入版本控制
- 后续不再需要把该文档当作当前执行入口

## 与 `docs/progress/` 的分工

- `docs/plans/` 记录“准备做什么、为什么这么做、如何验收”
- `docs/progress/` 记录“已经做了什么、验证结果是什么、还剩什么”

如果一份文档主要在陈述“本次实施结果”“验证输出”“剩余事项”，它通常更适合进入 `docs/progress/`。
