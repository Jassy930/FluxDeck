# FluxDeck macOS Native 原生界面重设计

## 目标

将 `apps/desktop-macos-native` 从当前偏 MVP 管理面板的原生壳，升级为强对齐参考图风格的 macOS 桌面控制台。

本次设计采用：

- `B 中改信息架构`
- `全部页面统一风格`
- `强对齐参考图`

核心要求不是简单“换皮”，而是在保留现有 Admin API 数据流与业务能力的前提下，重做应用壳、导航结构、首页监控工作台与拓扑主画布。

## 产品定位

产品第一视觉应从“资源列表管理器”转变为“深色专业网络控制台”。

用户打开应用后的第一判断应该是：

1. 当前系统是否健康
2. Gateway / Provider 是否运行正常
3. 最近请求流量、错误与路由关系是什么

因此视觉与信息层级需要从“对象管理优先”转为“监控台优先，资源管理次之”。

## 总体架构

整体界面分为三层：

### 1. 左侧分组导航

采用参考图中的分组式 source list 结构，而不是当前纯平的系统默认 sidebar。

建议分组如下：

- `Overview`
- `Traffic`
- `Connections`
- `Logs`
- `Visualization`
  - `Topology`
  - `Route Map`
- `Proxy`
  - `Providers`
  - `Gateways`
- `System`
  - `Settings`

设计目标：

- 具备专业工具气质
- 分组清晰，密度较高
- 选中态克制，不使用夸张高亮

### 2. 顶部模式条

在主内容区顶部加入强对齐参考图的 segmented control，用于承载全局模式感知。

建议模式项：

- `Backup`
- `Direct`
- `Rule`
- `Global`

第一阶段允许其作为 UI 模式状态与未来路由模式的预留入口存在，不要求立刻绑定全部真实后端逻辑。

顶部右侧同时放置轻量状态徽标：

- Admin API 连接状态
- Gateway 运行汇总状态
- 最近错误 / 告警入口

### 3. 主内容画布

所有页面共享统一的内容区结构：

- 页面标题与操作区
- 页面级筛选 / 状态条
- 主卡片区
- 次级分析区或详情区

不再让系统默认 `List` 充当页面骨架主角。

## 视觉系统

整体风格：`深石墨 + 深色磨砂 + 柔和描边 + 彩色状态点缀`

### 背景与层次

- 主背景使用近黑石墨色，不使用纯黑
- 外层工作台为大圆角深色容器
- 内层卡片为半透明深灰磨砂卡
- 层级主要通过明度差、描边与留白建立，而非重阴影

### 颜色体系

- 主背景：`#131416` / `#17181B`
- 卡片底色：`#1B1D21` / `#23262B`
- 边框描边：`#2B2F36`
- 主文字：`#F3F4F6`
- 次文字：`#9AA0AA`
- 弱文字：`#6F7682`
- 强调青蓝：`#3CCBFF`
- 强调绿色：`#4ADE80`
- 强调橙色：`#FF9F43`
- 强调洋红：`#C061FF`
- 错误红：`#FF5D73`

### 圆角与控件

- 外层主容器：`24–28`
- 顶部模式条 / 分组容器：`14–18`
- 内容卡片：`16–20`
- 状态胶囊：极大圆角

整体圆角统一偏大，以接近参考图的柔和工业风。

### 字体与图标

- 保持 `SF Pro` 系统字体
- 大数字与指标强调桌面工具的稳定感
- 图标统一使用 `SF Symbols`
- 默认低饱和，选中态或状态态才提高可见度

### 动效原则

- 所有微动效控制在 `180–240ms`
- 不使用持续炫技动画
- 仅在刷新、切换、hover、状态更新时做轻量过渡
- 尊重 `Reduce Motion`

## 页面设计

## 1. Overview

定位：`监控首页 / 运行驾驶舱`

第一屏布局：

- 第一行：`Running Status`、`Network Status`
- 第二行：`Traffic Stats`、`7-Day Traffic Trend`
- 第三行：`Traffic Summary`、排名 / 分布面板

目标：

- 让用户第一眼看到系统健康度
- 用现有 `providers / gateways / logs` 推导出监控型信息
- 将首页从“欢迎页”升级为“控制台主舞台”

## 2. Topology

定位：`独立主画布 / 路由结构理解页`

页面结构：

- 顶部控制条：时间窗口、布局方式、刷新
- 中间大画布：`Entrypoints -> Gateways -> Providers -> Models` 或简化三层结构
- 侧边 / 底部详情卡：展示节点与链路信息

编码规则：

- 节点颜色区分类型
- 连线粗细表达请求量
- hover 只高亮相关链路
- 点击节点后在详情卡中展示统计信息与错误摘要

第一阶段重点是形成参考图 2 的画布感与层级关系，不追求复杂实时引擎。

## 3. Traffic

定位：`流量分析页`

建议内容：

- 顶部 KPI：请求量、成功率、P95 延迟、活跃 Gateway
- 中部趋势图：按时间维度展示请求与延迟走势
- 底部分布图：Provider / Gateway / Model 维度切换

## 4. Connections

定位：`连接与活跃会话页`

第一阶段基于现有日志数据做近似表达：

- 活跃入口
- 最近活跃 Gateway
- Provider / Model 命中情况
- 连接类摘要卡片

## 5. Logs

定位：`分析型日志工作台`

结构建议：

- 顶部筛选胶囊：Gateway、Provider、Status、Errors only、时间范围
- 主区：日志表
- 侧区：选中日志详情卡

从“单纯列表”升级为“可筛选、可钻取、可分析”的监控式日志页。

## 6. Providers

定位：`资源管理工作台`

结构建议：

- Provider 概览卡片网格或主列表
- 右侧详情区展示配置摘要与状态
- 创建 / 编辑保留 sheet，但页面外壳统一新视觉体系

## 7. Gateways

定位：`运行态与配置并重的网关工作台`

视觉重点：

- 强化 runtime status
- 强化 listen host / port
- 强化 last error
- `Start / Stop` 使用明显但克制的状态按钮

## 8. Settings

定位：`系统配置面板`

建议分区：

- `Admin API`
- `Refresh & Sync`
- `Diagnostics`

每个分区都采用卡片化容器，不退回默认系统设置页质感。

## 交互规则

- 侧栏点击即切页，不走层层 push 导航
- 顶部模式条采用稳定 segmented control 交互
- 卡片 hover 只做轻量提亮与描边增强
- Overview 指标卡支持 drill-down 到详情页
- Topology 点击节点后更新详情区，而不是频繁弹窗
- 所有异步操作都必须有 loading / disabled / success / error 反馈

## 状态设计

### 加载态

- Overview 使用 skeleton 卡片而不是空白页
- Topology 使用节点骨架和轻量画布加载提示
- 列表页使用卡片骨架替代单个小 spinner

### 空态

- 无 Provider：引导创建 Provider
- 无 Gateway：引导创建并启动 Gateway
- 无 Logs：说明当前尚无流量

### 错误态

- 全局错误放在顶部细条提示
- 页面级错误留在卡片内部并提供 `Retry`
- Gateway 错误需要在详情区持续可见

## 实现原则

- 保留 `AdminApiClient`、模型、请求与提交流程
- 重构 `ContentView`，将当前大文件拆解为统一壳层与页面模板
- 优先建立视觉组件系统，避免每页各写一套样式
- 第一阶段不新增后端契约，不引入复杂拓扑引擎

## 非目标

- 不在本阶段实现复杂实时拓扑动画系统
- 不在本阶段引入新的后端数据契约
- 不为视觉改造而重写业务逻辑
- 不退回系统默认表单页与列表页风格

## 验收标准

1. 打开应用第一眼接近参考图的深色桌面控制台气质。
2. 所有页面视觉统一，不再出现明显的系统默认页面风格。
3. `Overview` 成为真正的监控首页，`Topology` 成为独立主画布。
4. `Providers / Gateways / Logs / Settings` 统一套入新工作台风格。
5. 现有 Provider / Gateway / Logs 管理能力保持可用。
6. 后续实现能够按组件和页面逐步推进，而无需整体推翻。
