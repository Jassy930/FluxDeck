# FluxDeck 前端并行方案设计文档

## 1. 背景与目标

当前项目目标是尽快发布 MVP，同时验证 macOS 原生前端路线可行性。为兼顾速度与长期体验，采用“双线并行、单后端契约”的方式推进：

- 主线：Tauri 作为首发版本 UI
- 支线：SwiftUI 原生壳并行验证
- 共享后端：`fluxd` Admin API 为唯一业务入口

## 2. 并行交付架构

### 2.1 进程与职责

- `fluxd`：控制面与数据面统一后端
- `fluxctl`：CLI 管理入口
- `apps/desktop`：Tauri 主线发布前端
- `apps/desktop-macos-native`：SwiftUI 原生壳（探索线）

### 2.2 共享契约原则

- 两个前端都通过 HTTP 调用 `fluxd` Admin API
- 不允许前端直接读写 SQLite
- 后端业务逻辑只保留在 `fluxd`

## 3. 里程碑与排期

### M1：首发就绪（优先级最高）

- 完成 Tauri 可用管理界面（Provider/Gateway/Logs）
- 完成主线稳定性与错误处理补强
- 通过发布验收命令

### M2：原生壳并行验证（不阻塞首发）

- 建立 SwiftUI 工程并可启动
- 打通 Provider/Gateway 列表读取
- 提供创建入口占位与错误提示

### M3：首发后路线决策

- 根据反馈决定：持续增强 Tauri 或推进原生迁移
- 若推进原生，再逐步补 CRUD、启停与日志能力

## 4. 并行开发边界

### 4.1 主线（Tauri）

- 可改范围：`apps/desktop` 下 UI 与状态管理
- 禁止：复制后端业务逻辑到前端

### 4.2 支线（原生）

- 可改范围：`apps/desktop-macos-native` 独立目录
- 禁止：绕过 `fluxd` 直接访问数据库

### 4.3 契约变更规则

- 后端接口新增可直接扩展
- 删除/重命名字段需保留兼容期
- 契约变更必须先更新文档，再同步两端适配

## 5. 测试与验收矩阵

### 5.1 主线发布必过

```bash
cargo test -q
bun run test --cwd apps/desktop
./scripts/e2e/smoke.sh
```

### 5.2 原生壳必过

- Xcode 工程可构建启动
- 可读取并展示 Provider/Gateway 列表
- 网络失败时 UI 可见错误，不崩溃

### 5.3 发布判定

- 主线全绿即可发布
- 原生壳问题不阻塞首发，但需登记问题清单

## 6. 当前结论

- 决策：采用并行方案，主线优先发布，原生壳同步搭建。
- 目标：在不牺牲发布速度的前提下，尽早验证原生演进路径。
