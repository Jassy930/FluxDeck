# FluxDeck Native Logs Density Polish 设计

## 文档状态

- 状态：已确认
- 类型：设计文档，待进入实现
- 适用范围：`apps/desktop-macos-native`

## 背景

当前原生端 `Logs` 页面已经完成从双栏详情页到单列可展开日志流的重构，但列表主体仍然带有明显的“大卡片”视觉节奏：

- 单条日志上下留白较多，连续扫描效率不足
- 折叠态主信息分布偏松，像卡片摘要，不像运维工作台里的日志行
- 顶部 `Log Filters` 仍然是一张相对厚重的内容卡，与下面的日志流风格不够统一

用户希望将日志页继续收敛为更像原生控制台的高密度日志视图，并明确要求：

- 日志列表更接近“表格行”而不是松散卡片
- 折叠态保留两行紧凑结构
- 首屏直接展示 `状态 + 路由 + 模型/错误 + tokens + latency + time`
- 顶部筛选区与日志列表使用统一的紧凑风格

## 已确认产品决策

- 日志页延续单列可展开工作台，不回退到左右分栏
- 列表视觉从“大卡片”收敛为“行式日志流”
- 折叠态采用“两行紧凑”结构，而不是单行极限截断
- 折叠态核心字段优先级固定为：
  - `状态`
  - `路由`
  - `模型 / 错误摘要`
  - `token 摘要`
  - `latency`
  - `time`
- 顶部 `Log Filters` 与下方日志列表统一为更轻、更扁平的运维工具风格

## 目标

1. 让用户在同样高度中看到更多日志项，提升扫描密度
2. 让折叠态直接承载请求定位与初步诊断所需的主信息
3. 保持失败日志足够醒目，但不破坏整个列表的连续阅读节奏
4. 统一筛选区与日志流的视觉语言，减少页面内部风格割裂

## 非目标

- 不调整后端日志契约与字段口径
- 不新增全文搜索、时间范围筛选、排序、分组或导出能力
- 不把日志流彻底重构为 SwiftUI `Table`
- 不改动 Overview、Traffic、Connections 等其他页面

## 设计方案

### 1. 顶部筛选区改为紧凑工具栏

保留现有筛选能力，但改变视觉语义：

- 从“独立大卡片”改为“日志流控制条”
- 保留字段：
  - `Gateway`
  - `Provider`
  - `Status`
  - `Only Errors`
  - `Loaded x`
  - `More available`
  - `Clear Filters`

布局建议：

- 左侧为三个 `Picker` 与 `Only Errors`
- 右侧为加载状态标签与 `Clear Filters`
- 标题弱化，不再占据明显首行高度
- 内边距、控件高度和圆角统一下调，整体更贴近工具栏而不是内容卡

### 2. Request Stream 改为行式日志流

保留 `Request Stream` 作为主体容器，但内部日志项改成“近似表格行”的连续列表：

- 行与行之间使用窄间距与细分隔
- 弱化每条日志独立卡片的厚重感
- 保留点击整行展开的交互
- 保留一次只展开一条日志的手风琴逻辑

实现上不强制切换到 `List`，但视觉效果应接近原生监控台中的高密度操作流。

### 3. 折叠态使用两行紧凑布局

#### 第一行：主判断层

第一行用于快速判断请求去向与结果：

- `状态码 pill`
- `gateway -> provider`
- `模型 / 错误摘要`

规则：

- 成功日志默认突出模型信息
- 失败日志优先突出错误摘要，模型退为次级信息
- 路由保持单行
- 模型或错误摘要允许有限截断，避免把行高拉高

#### 第二行：诊断上下文层

第二行用于提供紧凑上下文：

- `tokens` 摘要
- `latency`
- `time`
- `protocol / stream` 小标签

规则：

- 第二行整体字号更小、颜色更弱
- 数字字段可采用局部等宽，提升纵向扫描效率
- 时间精度保留到秒，避免折叠态宽度被毫秒时间戳挤占

### 4. 成功 / 失败状态的视觉策略

不使用大面积红底或强烈背景切换，而采用局部强化：

- 成功日志：中性背景、弱描边
- 失败日志：更清晰的语义描边或左侧强调线
- 错误摘要使用错误语义色
- 成功日志保持克制，避免整个列表过度噪音

这可以让失败项“跳出来”，但不会打断连续扫读。

### 5. hover / expanded 交互策略

折叠态与展开态之间保持平滑而稳定的过渡：

- `hover`：背景略提亮、边框略增强
- `expanded`：当前行底色上提，展开区贴合在同一行下方
- `chevron`：仅做方向旋转
- 不使用 `scale` 类动画，避免列表抖动

### 6. 展开态改为分组式诊断区

展开态仍然内嵌在当前日志项中，但结构从“逐条 detailRow”收敛为有分组的诊断区：

- `Execution`
  - `request_id`
  - `protocol`
  - `stream`
  - `first_byte`
- `Diagnostics`
  - `input / output / cached / total tokens`
  - `error_stage`
  - `error_type`
  - `error`
  - `usage_json`

布局建议：

- 短字段使用双列紧凑键值块
- 长文本如 `error`、`usage_json` 单独占满一行
- 视觉上仍属于“日志行展开部分”，而不是新的详情卡

## 文案与格式化规则

### 折叠态

- `route`：单行显示，过长时尾部截断
- `summary`：成功时优先模型；失败时优先错误摘要
- `tokens`：使用稳定短格式，例如 `Tok 1.2k in / 3.4k out / 320 c`
- `latency`：保持 `1234 ms`
- `time`：显示到秒级，保留等宽数字风格

### 展开态

- `request_id` 与 `usage_json` 允许复制
- 长错误文本完整显示
- 不在展开态重复折叠态已足够明确的主信息

## 组件与实现建议

优先在 `LogsWorkbenchView.swift` 内完成本轮收敛，避免过度拆分；若结构开始变得臃肿，可抽离以下私有子视图：

- `LogToolbar`
- `LogRow`
- `LogRowMetaStrip`
- `LogExpandedSection`

本次重点是密度与信息层级调整，不要求为了组件拆分而增加额外复杂度。

## 测试要求

1. 保持现有日志展开状态机行为不变：
   - 同一时刻仅一条展开
   - 切换筛选后收起旧展开项
   - 日志列表变化后清理失效展开项
2. 为新的折叠态摘要模型补测试：
   - 失败日志优先错误摘要
   - 折叠态直接包含 token 摘要、latency、time
   - 成功与失败日志格式化结果不同
3. 为展开态分组文案补测试：
   - `Execution` 与 `Diagnostics` 所需字段完整可达
   - `usage_json` 为空时不渲染原始明细块

## 验收标准

1. 日志列表视觉明显比当前更紧凑，一屏可见日志数量增加
2. 顶部 `Log Filters` 与日志列表风格统一，不再像独立厚重卡片
3. 折叠态两行内可直接读到 `状态 / 路由 / 模型或错误 / tokens / latency / time`
4. 失败日志更容易识别，但不会造成整页大面积语义色污染
5. 展开态仍在当前日志项内完成诊断查看，不引入新的详情区域
